# 高层级切片参数测试报告

测试日期：2026-04-25

测试对象：
- 项目：`freestiler`
- 数据源：`PostGIS`
- 输出：`MongoDB`
- 真实大表：`public.ht_tyg5c32ihg_sys_ht_mark`
- 估算行数：`2548805`

测试目标：
- 验证高 zoom 下 `PostGIS -> Mongo` 的真实可用性
- 总结高层级切片的推荐参数
- 明确高层级区间与低层级区间的风险边界

## 测试环境

PostGIS 连接：

```text
10.1.0.16:5433:geoc_data:postgres:postgres
```

Mongo 连接：

```text
localhost:27017
```

核心固定参数：
- `tile_format = Mvt`
- `simplification = true`
- `coalesce = false`
- `drop_rate = None`
- `cluster_distance = None`
- `cluster_maxzoom = None`
- `PostgisConfig.batch_size = 10000`
- `PartitionConfig.partition_zoom = max_zoom`
- `PartitionConfig.metatile_rows = 64`
- `MongoSinkConfig.batch_size = 4096`
- Mongo 文档字段：`id / x / y / z / data`

## 测试方法

真实一致性测试：
- 使用 `postgis_to_mongo_matches_reference_tiles`
- 对比原生参考切片、insert、upsert、streaming 三条路径
- 校验 tile key 集合、tile 内容摘要、重复写入、Mongo 文档字段形状

读取性能测试：
- 使用 `postgis_large_table_reader_profile`
- 只比较旧 reader 与优化后 reader 的分区读取耗时
- 该测试是 reader 采样对比，不等于完整 end-to-end 总耗时

## 高层级性能观察

Reader 采样性能：

| zoom | old_ms | new_ms | speedup |
| --- | ---: | ---: | ---: |
| `z8` | 21002 | 16558 | 1.27x |
| `z9` | 21129 | 16578 | 1.27x |
| `z10` | 9278 | 4783 | 1.94x |
| `z11` | 6612 | 33 | 197.11x |
| `z12` | 6492 | 21 | 304.89x |

说明：
- `z10` 以后，单个分区覆盖的空间范围明显变小，reader 的 bbox 过滤收益迅速放大。
- `z11`、`z12` 的 reader speedup 很高，主要是因为采样分区变得非常稀疏，优化 reader 读取到的数据量急剧下降。
- 这个 speedup 只能说明“读取阶段”受益很大，不能直接等同于完整切片总耗时也有同等倍率提升。

真实端到端耗时样本：

| 配置 | 耗时 |
| --- | ---: |
| `z12` 单层 | 286.46s |
| `z11` 单层 | 709.24s |
| `z10` 单层 | 268.98s |
| `10..12` 区间 | 1344.08s |

说明：
- 高 zoom 下总耗时不只受 reader 影响，还受切片数量、编码、Mongo 写入吞吐影响。
- `z11` 比 `z12` 更慢，说明真实端到端时间还会受当前层级的空间分布和 tile 数量影响，不能只看 zoom 数字线性推断。

### `14+` 单链路 streaming 性能样本

说明：
- 这里不是全量 reference 一致性联调，而是只测试 `run_postgis_to_mongo_stream` 单链路。
- 目的不是再做一次超长全量比对，而是确认 `14` 级及以上在真实环境下能否完成，以及耗时大致落在哪个量级。

| 配置 | tile 数 | 耗时 |
| --- | ---: | ---: |
| `z14` 单层 streaming | 196031 | 288.28s |
| `z15` 单层 streaming | 762740 | 771.16s |
| `z16` 单层 streaming | 1983370 | 1730.08s |

结论：
- `z14` 可以完成，且单链路耗时大约 `4.8` 分钟。
- `z15` 也可以完成，但单链路耗时已经上升到大约 `12.9` 分钟。
- `z16` 仍然可以完成，但单链路耗时已经到大约 `28.8` 分钟。
- `z15` 的 tile 数相对 `z14` 明显放大，说明 `14+` 的主要成本已经转移到 tile 数量、编码和 Mongo 写入，而不是 PostGIS 读取。
- `z16` 的 tile 数进一步放大到接近 `200` 万，说明 `16+` 已经不是默认生产区间，而是明确的高成本区间。
- `z14` 的全量一致性联调之所以超过 2 小时，不代表它不能跑，而是因为 reference / insert / upsert / streaming 四条链路一起比对太重。

## 参数结论

### 推荐参数

如果目标是高层级切片并写入 Mongo，推荐参数为：

```text
tile_format = Mvt
min_zoom = 10
max_zoom = 12
simplification = true
coalesce = false
drop_rate = None
cluster_distance = None
cluster_maxzoom = None
postgis.batch_size = 10000
partition_zoom = max_zoom
metatile_rows = 64
mongo.batch_size = 4096
```

推荐理由：
- `10..12` 已完成真实联调，结果与原生一致。
- 该区间完全避开低 zoom 超大 tile 风险。
- `partition_zoom = max_zoom` 符合当前流式架构，能够最大化高 zoom 的空间裁剪收益。
- `metatile_rows = 64` 在当前真实大表下已经验证稳定，不需要额外调大或调小。

### `14` 级及以上建议

如果业务明确需要更高层级，推荐这样使用：

```text
min_zoom = 14
max_zoom = 16
postgis.batch_size = 10000
partition_zoom = max_zoom
metatile_rows = 64
mongo.batch_size = 4096
```

使用原则：
- `14+` 不建议作为默认全量区间。
- 更适合作为按需开启的高精度区间。
- `z14` 可接受，`z15` 已明显更重，`z16` 已进入接近半小时级别的单链路处理时间。
- 如果业务要稳定响应，建议把高精度区间控制在 `14..15`。
- `z16` 可以作为离线批处理区间，但不建议作为日常默认参数。
- `17+` 当前还没有真实完整结果，不建议直接默认打开。

### 保守生产参数

如果希望兼顾稳定和覆盖范围，推荐：

```text
min_zoom = 6
max_zoom = 12
```

说明：
- 当前真实大表下，`z6` 是 Mongo 可落地的最小安全单层 zoom。
- 只要区间包含 `z5`，就已经出现真实超限。
- 所以对这张表，Mongo 输出的安全下限应视为 `min_zoom >= 6`。

### 不推荐参数

以下参数区间当前不建议用于 Mongo 输出：

```text
min_zoom <= 5
```

原因：
- 这不是“性能可能差一点”的问题，而是会直接触发 Mongo `16MB` 单文档上限。
- 当前真实大表已经验证：
  - `z5` 超限
  - `0..8` 超限

## 架构结论

- 流式架构适合这个项目。
- 当前瓶颈的本质不是 streaming，而是低 zoom 下单 tile 聚合密度过高。
- 对 MLT/MVT 这类最终编码格式，是否超限只能在 tile 编码完成后得到确定答案，所以“完全编码前预估”并不可靠。
- 当前实现采用“编码完成后，进入 sink 前预校验；sink 末端再保底”的方式，是比较符合原生架构的做法。

## 最终建议

- 高层级切片直接采用 `10..12` 作为默认推荐区间。
- 如果业务必须覆盖更低层级，Mongo 输出时不要低于 `z6`。
- 如果业务必须输出 `z5` 及以下，建议不要继续走 Mongo 单文档存储方案，而应考虑换存储载体，或者对源数据先做更强的聚合与裁剪。
